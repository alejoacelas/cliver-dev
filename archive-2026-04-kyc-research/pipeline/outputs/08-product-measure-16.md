# Measure 16 — MFA Step-Up: Product Prioritization

## Selected stack

### 1. Hosted IdP MFA + step-up (`m16-auth0-okta`)

Selected as the foundation. This is the delivery mechanism for the entire measure: it provides the OIDC plumbing, factor enforcement policy, adaptive risk scoring, and step-up endpoint that every other selected idea depends on. Pluggability is high — Auth0, Okta CIC, and AWS Cognito are all turnkey hosted IdPs that a third-party screening service could pre-integrate, giving providers a drop-in OIDC integration rather than bespoke infrastructure. The interface contract is clean: the order pipeline calls a step-up endpoint, the IdP returns `auth_time`, `acr`, and `amr` claims that the pipeline validates. Attacker-story coverage is the broadest of any single idea (8 of 12 mapped bypasses caught when configured for passkey-only with IDV-gated recovery). Marginal cost per step-up is $0 with passkey/TOTP. The main cost is platform fee and the 4-8 engineer-week integration, but this is a one-time investment that all other M16 ideas build on top of.

### 2. Order-time `max_age=0` step-up (`m16-order-time-stepup`)

Selected because it closes two bypass methods that no other idea addresses: session hijack via stolen cookies and grace-period exploitation. Without this, an attacker who steals a session cookie can place SOC orders without ever touching the MFA layer. The check is trivially pluggable (a single OIDC parameter on the authorize redirect), has zero marginal cost, and takes 1-2 engineer-weeks to implement. The interface is as clean as OIDC gets: add `max_age=0` and `acr_values` to the redirect, validate `auth_time` on the returned token. It composes perfectly with the hosted IdP — it is literally a parameter on the same endpoint. The programmatic/API ordering gap (10-30% of volume) is a real concern that the downstream product spec must address, but for browser-based orders this is a low-cost, high-value addition.

### 3. No-SMS, no-email-reset SOP (`m16-no-sms-no-email-reset`)

Selected because it closes the single most exploited recovery-channel bypass — email-based MFA reset — which is the common entry point across account-hijack, credential-compromise, and dormant-account-takeover attacker stories. Without this SOP, the hosted IdP's passkey enforcement is undermined: an attacker who compromises the customer's email can reset MFA, enroll their own factor, and pass step-up cleanly. This is a configuration + policy idea with $0 configuration cost. The operational cost ($5-15 per recovery event, 500-1000 tickets/year for a 10K-customer provider) is the price of closing the email-channel bypass — and it is modest relative to the attack surface it eliminates. It composes naturally: it is an IdP configuration enforced on the same platform selected in idea 1, and its 24-hour cooling-off period cross-references the order-time step-up in idea 2.

### 4. Dormancy re-IDV trigger (`m16-dormancy-reidv`)

Selected because it addresses a structurally distinct attack branch — dormant account takeover by institutional IT administrators — that no other idea covers. The selfie-match-against-stored-portrait design is uniquely suited to catching a different-person takeover, which password-based and factor-based checks cannot detect. The marginal cost is low (~$1.50-3.00 per re-IDV event, <$0.20/customer/year amortized) and the false-positive burden is bounded (3-7% of academic customers annually, ~3 minutes each). Setup is 1-3 engineer-weeks on an existing IdP. The check is factor-agnostic — it asks "are you the same human?" — making it orthogonal to the factor-type controls in ideas 1-3 and providing a layer of defense that survives even if all MFA controls are bypassed on a dormant account.

## Dropped ideas

- **WebAuthn / YubiKey hardware token enforcement (`m16-webauthn-yubikey`):** Dominated by `m16-auth0-okta` with passkey-only policy. The AAGUID allowlist adds marginal security against cloud-passkey injection, but at prohibitive operational cost ($50-100/customer hardware, international distribution logistics, 10-20% onboarding gap, import restrictions in ~20 countries). The hosted IdP already enforces FIDO2/WebAuthn; mandating hardware-only tokens trades coverage breadth for a narrow security gain that most providers will not accept.

- **SpyCloud / Constella breach-credential check (`m16-spycloud-breach`):** Low incremental value for M16. Breach-credential detection is a credential-hygiene control, not an MFA step-up control — it catches compromised passwords, not MFA bypasses. When passkey-only enforcement is in place (idea 1), password compromise alone cannot complete step-up. The 30-50% email-only hit rate in academic populations would generate unsustainable review noise. Vendor-gated pricing (low-five to low-six-figure annual contracts) for a backward-looking check that structurally misses real-time AitM phishing. Better suited as a standalone credential-hygiene measure than as part of M16.

## Composition note

The four selected ideas form a layered defense that operates on the same IdP platform and shares a single OIDC integration point. At the infrastructure layer, the hosted IdP (idea 1) provides the authentication engine, factor policy enforcement, adaptive risk scoring, and audit log that all other ideas consume. At the order layer, the `max_age=0` step-up (idea 2) is a parameter on the same OIDC authorize endpoint, ensuring every SOC order is bound to a fresh authentication event. At the recovery layer, the no-SMS/no-email-reset SOP (idea 3) is an IdP configuration that hardens the factor lifecycle against the cheapest takeover path. At the account-lifecycle layer, the dormancy re-IDV trigger (idea 4) fires as a post-login Action in the same IdP, routing dormant accounts through the existing IDV vendor before they can place orders.

All four ideas run sequentially in a single authentication flow: dormancy check fires first (is this account dormant? → force re-IDV), then the step-up redirect fires at order time (`max_age=0` with passkey-required `acr_values`), and the recovery SOP governs what happens when factor reset is needed at any point. Their outputs combine into a single per-order audit record: `auth_time`, `acr`, `amr`, dormancy-reidv status, and any flags (`mfa_not_enrolled`, `stepup_failed`, `dormancy_reidv_selfie_mismatch`, etc.) route to one reviewer queue. The shared inputs are the customer's `user_id` and `last_login_at` timestamp (for dormancy), plus whatever the IdP already knows about enrolled factors.

Integration concerns for the downstream product spec: (1) the programmatic/API ordering gap identified in idea 2 needs a parallel authentication mechanism for LIMS integrations, likely OAuth client-credentials with per-request signing; (2) the federated SSO path (15-25% of academic customers) requires provider-side `amr` validation and possibly a provider-side passkey requirement independent of the upstream IdP; (3) the passkey enrollment ramp at rollout (30-60% of customers may lack passkeys) needs a phased enforcement plan with a TOTP grace period and enrollment nudges before hard cutover.
