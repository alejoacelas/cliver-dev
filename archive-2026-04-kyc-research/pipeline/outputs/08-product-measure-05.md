# M05 — Shipping-Institution-Association: Product Prioritization

## Selected stack

### 1. m05-ror-gleif-canonical (ROR / GLEIF / Companies House canonical address cross-reference + OSM campus polygon augmentation)

This is the foundational automated check for M05 and should be implemented first. It scores highest on pluggability (clean API-based inputs, no scraping), interface clarity (structured resolution cascade with labeled confidence levels), coverage breadth (three registries covering research organizations, LEI-registered entities, and UK companies, augmented by OSM polygons and GeoNames for geographic containment), and composability (its resolution-level output feeds directly into downstream escalation logic). The resolution cascade design — from street-level GLEIF/Companies House matching down through OSM polygon containment, GeoNames point-radius, and city-level fallback — provides graceful degradation rather than binary pass/fail, which is critical for a measure that spans diverse institution types globally. At <$0.02/check marginal cost with no per-API authentication barriers for the core registries, the false-positive burden is managed by surfacing the resolution level to reviewers. The 2-3 engineer-week setup cost is justified by the breadth of attacker stories addressed (ghost-office, address-spoofing, shell-company, and with the OSM augmentation, same-city CMRA and virtual-office bypasses). Its main coverage gaps — entities not in any registry (10-20%), non-OECD institutions with poor OSM coverage, and multi-campus false positives — are partially addressable by the other selected ideas.

### 2. m05-two-contact-sop (Two-contact directory verification SOP)

This is the essential escalation backstop and should be implemented as the manual review layer behind the automated checks. It scores highest on attacker-story coverage among the four ideas: it catches shell entities, virtual offices, fabricated institutions, and CRO-framing paths that no automated registry or polygon check can detect, because it tests whether a real institution with real people stands behind the claimed affiliation. The `two_contact_independence_failure` flag — firing when no public directory exists at all — is the single strongest negative signal in the M05 measure and catches attacker stories that all three other ideas miss entirely. Its interface clarity is strong (structured five-step SOP with explicit disposition rules), and it produces the best audit trail in the pipeline (human-to-human attestation with domain-verified email replies). The $30-60/case cost and ~6-10 cases/day/FTE throughput make it viable only for borderline escalations, which is exactly where it belongs: behind the ROR/GLEIF automated check, receiving only the cases that fail or produce ambiguous automated results. The composability is excellent — it consumes the resolution-level output from m05-ror-gleif-canonical to decide which cases warrant the investment.

## Dropped ideas

### m05-google-places-campus (Google Places + OSM campus polygon containment)
Dropped: duplicates the OSM polygon containment already built into m05-ror-gleif-canonical's resolution cascade, adds Google Places API cost ($0.032/check) and ToS risk without meaningfully expanding coverage beyond what ROR + OSM already provide. The incremental value over the selected stack is near zero — the only addition is using Google Places for institution resolution, but ROR already handles this for research organizations, and the polygon logic is identical.

### m05-incubator-tenant (Incubator / coworking tenant directory)
Dropped: extremely narrow firing surface (<1% of orders), high false-positive rate among triggered cases (10-30%), fragile scraping-based implementation with linear maintenance burden, and the structural limitation that the primary attacker it targets (biotech-incubator-tenant) defeats the check by paying for a real lease. The cost-raising value is real but marginal relative to the engineering and operational investment. If biotech-incubator orders become a significant concern, this could be revisited as a lightweight address-denylist (flagging known incubator addresses for extra scrutiny via the two-contact SOP) rather than a full tenant-directory scraping pipeline.

## Composition note

The two selected ideas form a layered automated-then-manual pipeline. Every order runs through m05-ror-gleif-canonical first: clear matches at street or polygon level pass; clear mismatches (cross-city, cross-country) are flagged with high confidence. The ambiguous middle — city-level-only matches, `institution_no_canonical_record` flags, polygon-edge cases, and multi-campus mismatches — escalates to m05-two-contact-sop for human verification. This composition directly addresses the main weakness of each idea in isolation: the automated check lacks judgment for edge cases, and the manual SOP lacks the throughput for bulk screening. The two-contact SOP should also re-fire on any in-system shipping-address change (flagged as an open issue in both syntheses). Residual gaps that this stack cannot address — carrier-level post-shipment redirects, inside-institution attackers with genuine affiliations, and industry customers with no registry presence — should be noted as requiring controls from other measures (M18/M19 legitimacy checks, carrier-level controls).
