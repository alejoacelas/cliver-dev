## Section 2: Narrative

### What this check is and how it works

The ORCID OAuth check requires each voucher to prove control of a claimed ORCID iD through a standard 3-legged OAuth flow. The provider redirects the voucher to orcid.org, where they authenticate directly with ORCID; the provider receives a signed access token containing the verified ORCID iD, then pulls the voucher's employment, works, and person records via the ORCID API (v3.0). The employment records include disambiguated-organization identifiers (often ROR IDs), which bridge directly into the m20-ror-disjointness check for voucher-customer institutional independence. The Public API is free; the Member API ($1,250–$5,500/year) provides access to data the user marked "trusted parties only." Vouchers without ORCID are routed to an alternate-evidence SOP requiring institutional email DKIM plus at least one corroborating signal.

### What it catches

The check's primary value is in eliminating fabricated voucher identities. A manufactured IT persona, a dormant-domain fabricated voucher, a gig-platform recruit without a scholarly background, and an inbox-compromise attacker who only controls email (not browser sessions) all fail because they cannot produce a legitimate ORCID with populated employment and works records. The check also separates the provider's credential space from the voucher's identity space: compromising provider-side credentials does not give access to the voucher's ORCID account. For shell entities and CRO rotations, the check adds meaningful per-event friction — each new voucher must produce a credible ORCID or survive the alternate-evidence gauntlet.

### What it misses

The hardening stage identified one Critical finding: rubber-stamping by legitimate vouchers with real ORCIDs is systematically missed. ORCID OAuth proves that the voucher is who they claim to be and has a scholarly identity — it does not verify whether the voucher has reviewed the specific SOC order. Every attacker branch that uses a real, inattentive voucher (PI rubber-stamping, recruited co-conspirator, gradual-legitimacy cultivation, bulk-order noise cover) passes this check trivially. The alternate-evidence SOP is a secondary weakness: it provides a bypass safety valve for attackers who cannot produce an ORCID but can produce an institutional email plus a fabricated faculty page. The check also does not block self-vouching (orderer = voucher), does not specify ORCID re-authentication cadence (allowing post-onboarding account hijack), and faces a fundamental limitation in that ORCID profiles are self-asserted and unverified.

### What it costs

The Public API path is $0 marginal per check (token exchange + 1–3 record reads). Engineering setup is approximately 1–2 weeks for a standard OAuth + JSON ETL pipeline. The Member API ($1,250–$5,500/year) is worthwhile only if many vouchers keep their profiles private. Reviewer time is zero when ORCID OAuth completes successfully; ~10 minutes per voucher routed to the alternate-evidence path. The main cost driver is the alternate-evidence path volume: if ~20–40% of vouchers lack ORCID, reviewer burden scales accordingly.

### Operational realism

The OAuth flow is a standard web integration that researchers are increasingly familiar with — ORCID OAuth is already used by many publishers, funders, and institutional repositories. The record snapshot (employments, works, timestamps) provides a strong, replayable audit artifact. The critical operational gap is that the implementation does not specify ORCID re-authentication cadence: if OAuth is one-time-at-onboarding, an account hijack after onboarding bypasses the ORCID layer for all subsequent vouching actions. The hardening stage recommended per-action re-authentication for SOC orders above a risk threshold, plus PKCE in the OAuth flow to mitigate token replay.

### Open questions

The Critical finding C1 (rubber-stamping by real vouchers) is structural to ORCID as an identity check — no implementation tweak addresses per-order scrutiny. This requires a complementary check (per-order attestation, behavioral anomaly detection). The 4C claim check flagged that the direct-US ORCID membership price band ($1,250–$5,500/year) lacks a US-specific citation, and that the rate-limit figures (24/sec, 60-burst) are presented inside an `[unknown]` block inconsistently. The Google Workspace DKIM claim in the alternate-evidence SOP was flagged as overstated (requires admin action, not automatic). The coverage research flagged that ORCID's total registration count (14.7M from Aug 2022) is dated, and that the Spanish study on profile completeness may not generalize globally.

## Section 3: Open issues for human review

- **Critical hardening finding C1 (structural, surviving):** Rubber-stamping by legitimate vouchers with real ORCIDs is systematically missed. ORCID OAuth is an identity/affiliation check, not an order-scrutiny check. Every real-but-inattentive-voucher pattern passes. Requires complementary per-order attestation mechanism.
- **Moderate hardening finding M1 (surviving):** Alternate-evidence SOP is a bypass safety valve — institutional email + one corroborating signal is weaker than ORCID OAuth. Recommend tightening to DKIM-verified email AND at least two independent signals; flag alternate-evidence vouchers for elevated scrutiny.
- **Moderate hardening finding M2 (surviving):** ORCID OAuth token replay risk via AitM phishing. Recommend PKCE, provider-controlled redirect URI with HSTS, and per-action re-authentication.
- **Moderate hardening finding M3 (surviving):** ORCID re-authentication cadence unspecified. If one-time-at-onboarding, post-onboarding account hijack bypasses ORCID layer. Recommend annual re-auth minimum + per-action for high-risk SOC orders.
- **Moderate hardening finding M4 (surviving):** Self-vouching (orderer = voucher) not blocked. Recommend comparing `orderer_orcid` != `voucher_orcid` — trivial to implement.
- **[vendor-gated] field:** Direct US ORCID membership pricing — exact tier requires sales contact. Cited band ($1,250–$5,500/year) lacks US-specific source.
- **[unknown] field:** ORCID Public API rate limits — figures (24/sec, 60-burst) appear in tutorials but not cleanly sourced.
- **4C claim flag (MISSING-CITATION):** Direct US ORCID membership price band not substantiated by cited UK/Canadian sources.
- **4C claim flag (MIS-CITED):** Rate-limit numerics presented inside `[unknown]` block; should be either sourced or removed.
- **4C claim flag (OVERSTATED — minor):** ORCID-CA pricing "~$3,500 CAD/year" should be weakened to "low-thousands CAD/year, varies by tier."
- **Coverage Gap 5 (fabricated profiles):** ORCID does not verify self-asserted employment/works data. A careful attacker can create a convincing profile. Mitigation requires cross-checking against OpenAlex works and institutional ORCID member integrations.
- **No 06C (claim check on coverage) was run.** Coverage citations (Wikipedia ORCID, Springer survey, Toulouse study, Frontiers, ResearchGate) not independently verified.
