## Section 2: Narrative

### What this check is and how it works

This check uses a large language model (Claude or GPT) with structured-output mode to extract a canonical set of SOC-relevant fields from any free-text the customer provides — project descriptions, end-use notes, biosafety summaries, or voucher email bodies. The LLM produces a JSON object containing the intended use category, host organism, target gene, BSL level, select agent status, and IBC reference, along with confidence scores and verbatim quotes from the source text. A deterministic comparator then checks this extracted object against two independent sources: the customer's structured-form submission (m15-structured-form) and the sequence-screening output. Disagreements — such as the LLM extracting "protein expression in E. coli" from the free text while the sequence screening identifies Bacillus anthracis genes — generate flags at graded severity levels and route to human review. The LLM is doing extraction, not deciding legitimacy; the value comes from surfacing inconsistencies a human reviewer can then investigate.

### What it catches

The check is strongest against the most common and lowest-effort M15 evasion: vague or generic self-declarations that fail to acknowledge SOC content. When a customer writes "molecular biology research" or "research purposes" while ordering a select-agent gene, the LLM extraction yields null or generic values that disagree with the specific SOC identified by sequence screening, triggering `llm_sequence_disagreement`. This catches the lab-manager-voucher and unrelated-dept-student attack patterns directly. Inbox-compromise attacks that claim exemption from SOC disclosure are similarly caught — an exemption claim produces no substantive extraction while the sequence content shows a SOC. Bulk-order concealment using facility-wide boilerplate is partially caught when the generic declaration disagrees with specific SOC sequences in individual orders.

### What it misses

The check's fundamental limitation is structural: it detects inconsistency, not malice. When an attacker deliberately engineers the narrative to match the SOC content — as in the cro-framing, cro-identity-rotation, gradual-legitimacy-accumulation, and biotech-incubator-tenant stories — all three inputs (free text, form, sequence) agree, and no flag fires. Insiders and account hijackers who know the legitimate research scope write declarations that produce consistent extractions. These eight attacker stories (four purpose-built, four insider/hijacker) represent the alignment-by-construction bypass, which is not addressable within a consistency-checking approach. Additionally, ~15–30% of orders may contain terse or formulaic free text where the LLM cannot extract meaningful structured fields, reducing the check to a manual-review funnel. Non-English submissions (~10–20% of global orders) face degraded extraction quality, particularly in low-resource languages.

### What it costs

Marginal cost is very low: $0.005–$0.02 per order for 2–3 LLM calls, depending on vendor and model choice. Haiku 4.5 at $0.002 per call is the cost-optimized option; the batch API halves this further for asynchronous processing. Setup cost is 2–4 engineer-weeks for schema definition, prompt engineering, the deterministic comparator, an evaluation set, and reviewer integration. There is no per-vendor license fee beyond API usage. The dominant ongoing cost is reviewer time for adjudicating flagged orders, which depends on the false-positive rate — and the false-positive rate depends on the quality of customer-provided text.

### Operational realism

The reviewer's primary task on flagged orders is checking whether the LLM extraction is faithful to the customer's text (using the `extracted_quotes` field) or hallucinated. If faithful, the disagreement is real and the reviewer contacts the customer for clarification. If hallucinated, the reviewer discards the flag. This two-step triage — extraction fidelity first, then substance — keeps the reviewer focused on genuine disagreements. The audit trail includes the immutable LLM JSON output, the exact prompt and model version, the comparator diff, and the reviewer's adjudication. A key operational decision not yet resolved is whether extraction runs synchronously at order time or asynchronously via the batch API; the latter saves 50% on cost but delays flag generation.

### Open questions

The implementation does not include an extraction-accuracy evaluation benchmark, which the hardening review flagged as a Moderate finding — without knowing the hallucination rate on realistic customer text, the false-positive burden is unpredictable. The terse-submission coverage gap (15–30% of orders) has no citation and was flagged by the form check as load-bearing and under-sourced. The multi-category extraction problem (10–20% of academic orders) could be mitigated by allowing multi-label extraction, but this is not yet specified. The data-usage claim ("Anthropic API does not train on customer data by default") needs a direct citation to the commercial terms. The synchronous vs. batch processing decision affects both cost and operational latency but is unresolved.
