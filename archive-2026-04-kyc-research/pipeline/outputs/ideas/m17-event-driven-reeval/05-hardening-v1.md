# m17-event-driven-reeval — Bypass-Aware Hardening v1

- **measure:** M17 (pre-approval-list)
- **idea:** Event-driven re-evaluation of pre-approved entities

---

## Attacker Story Walk

### 1. gradual-legitimacy-accumulation

**Summary:** A patient attacker spends 6-12 months operating a real-but-hollow small biotech, building order history and entity signals, then places SOC orders once the entity looks "seasoned."

**Bypass methods relevant to this measure:**

- **Method 1 — Time-aged domain and clean order history.**
  - **Classification: MISSED** — the branch's entire design is to *not* trigger any event. The entity is real, the domain is aging normally, the orders are clean. No ownership change, no OFAC hit, no breach, no dormancy. The event-driven re-eval fires on *changes*; this branch generates *stability*. The implementation correctly acknowledges: "weakly addressed — the branch's whole point is not to trigger an event."

- **Method 2 — Aged domain auction purchase.**
  - **Classification: MISSED** — purchasing a domain does not create a corporate-filing event that OpenCorporates would surface. Domain transfers are visible in WHOIS but the implementation does not subscribe to WHOIS-change feeds.

- **Methods 3-10 (ORCID profile, publications, preprints, conference posters, Crunchbase listings, SBIR applications, LinkedIn, AI-assisted drafting).**
  - **Classification: MISSED** — none of these generate events in any of the four subscribed feeds (OpenCorporates, OFAC, SpyCloud, dormancy timer).

**Net assessment:** Essentially zero value against this branch. The branch is purpose-built to look like normal entity maturation. No corporate-event trigger fires because the entity operates as designed.

### 2. shell-company

**Summary:** Attacker builds or acquires a shell company with real employees to order SOC material.

**Bypass methods relevant to this measure:**

- **Bypass method 1 — Build entity signals from scratch.**
  - **Classification: MISSED** — same reasoning as gradual-legitimacy-accumulation. Building from scratch over 6-12 months generates no corporate events. LLC formation, EIN, DUNS, virtual office, website, bioRxiv preprints, and order-history building are all stable activities that produce no *changes* for OpenCorporates to surface (after the initial incorporation, which is not an event on an already-registered entity).

- **Bypass method 2 — Acquire an existing company.**
  - **Classification: CAUGHT** — acquiring a going concern creates corporate filing events (officer/director changes, registered-agent changes, possibly address changes). OpenCorporates' events endpoint would surface these. The implementation correctly identifies this: "the acquisition is a corporate filing, OpenCorporates surfaces it as an officer / control change, the pre-approval is demoted exactly when the new owner shows up."
  - **Uncertainty:** latency. OpenCorporates filing freshness varies by jurisdiction — UK Companies House is near-real-time; some US states are months behind. An attacker acquiring an entity in a slow-reporting state could act in the gap between the filing and OpenCorporates surfacing it.

- **Gradual order escalation + protocol framing.**
  - **Classification: MISSED** — no event is generated by gradually escalating from non-SOC to SOC orders. This is a behavioral pattern, not a corporate event.

**Net assessment:** Valuable against the acquisition variant but worthless against the build-from-scratch variant. The acquisition variant is the more expensive and riskier path for the attacker ($50K-300K+), so catching it is worthwhile but does not address the more common build-from-scratch approach ($10-30K).

## Findings

### Finding 1 — Critical: The build-from-scratch purpose-built-organization attack generates no events

- **Source:** gradual-legitimacy-accumulation (all methods), shell-company (method 1), biotech-incubator-tenant (method 4c), bulk-order-noise-cover (all methods).
- **Why missed:** The idea's four event feeds (OpenCorporates, OFAC, SpyCloud, dormancy) are all *change-detection* feeds. The purpose-built-organization branches are explicitly designed to operate without generating changes — they build *stable* entities from scratch. A newly incorporated LLC that operates normally for 12 months generates zero events on any of these feeds. The only event that *might* fire is the initial incorporation itself, but the implementation describes monitoring *existing pre-approved* entities, not screening *newly onboarding* ones (that's stage 4's job, not this idea's).
- **This is the canonical attack against pre-approval lists**, and this idea provides no signal against it.
- **Suggestion:** The gap is structural for an event-driven approach. Event-driven re-eval works for *drift* (M&A, sanctions changes, breaches) but not for *patient construction*. The only measure that addresses patient construction is a *periodic* re-evaluation that re-runs the full legitimacy check (m18, m19) on a calendar cadence — not just on trigger events. Consider adding a time-based re-evaluation trigger (e.g., annual re-verification of institutional legitimacy for all pre-approved entities, regardless of events) to complement the event-driven triggers.

### Finding 2 — Moderate: OpenCorporates jurisdiction latency creates a window for the acquisition variant

- **Source:** shell-company (method 2 — acquire an existing company).
- **Why missed:** The implementation acknowledges this under failure_modes_requiring_review ("corporate filings vary in publication latency by jurisdiction, from days to months"). For the acquisition variant, this means the attacker could acquire a company in a slow-reporting US state, use its pre-approved provider account to place SOC orders, and be done before OpenCorporates surfaces the ownership-change event.
- **Suggestion:** For the acquisition variant, the mitigation is the monthly full-reconciliation the implementation mentions under failure_modes. Make this reconciliation cadence explicit and shorter for entities with SOC-active accounts (e.g., weekly, not monthly).

### Finding 3 — Minor: OFAC fuzzy-match noise may desensitize reviewers

- **Source:** General.
- **Why missed:** The implementation notes "common names cause false hits; common in Russian / Arabic / Chinese transliterations" under false_positive_qualitative. High false-positive rates on OFAC fuzzy matching could create alert fatigue, causing reviewers to approve demotions routinely without deep investigation. This is a standard sanctions-screening problem, not specific to this idea.
- **Suggestion:** Tune the fuzzy-match threshold specifically for the pre-approved entity population (which is smaller and can tolerate tighter matching) rather than using the same threshold as initial screening.

### Finding 4 — Minor: Event-feed integration with m16-spycloud-breach and m16-dormancy-reidv is architecturally overlapping

- **Source:** credential-compromise, dormant-account-takeover.
- **Why missed:** Two of this idea's four event feeds (breach data, dormancy timer) are the same mechanisms as dedicated sibling ideas (m16-spycloud-breach, m16-dormancy-reidv). The implementation routes these through this idea's event router, but the actual detection and response logic is identical. This is not a bypass gap — it's an architectural note for stage 7 synthesis to avoid double-counting the coverage.
- **Suggestion:** Stage 7 should clarify that this idea's breach and dormancy feeds are wiring/orchestration contributions, not independent detection mechanisms.

## Verdict

**STRUCTURAL** — one Critical finding. The build-from-scratch purpose-built-organization attack (the canonical attack against pre-approval lists, used by 3 of 7 mapped stories) generates no events on any of this idea's four subscribed feeds. This gap is structural: event-driven re-evaluation fundamentally cannot catch an entity that operates stably. The gap is not addressable by tweaking the implementation — it requires a different *kind* of check (periodic calendar-based re-evaluation, not event-triggered). Pipeline continues but the finding is routed to human review at the end of the run.
